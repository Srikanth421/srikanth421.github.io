What is a Large Language Model?

⚙ Breaking Down "LLM"
Large

Billions of parameters. Trained on massive internet-scale datasets.

Language

Takes text in, produces text out. Understands human language patterns.

Model

A mathematical function that predicts the next token given previous tokens.

At its core, an LLM does one thing: given a sequence of words, predict the most likely next word. It does this with the Transformer architecture you already learned.

// What an LLM computes: P(next_word | "The cat sat on the") → "mat" (0.32), "floor" (0.21)... // Pick most probable, append, repeat: "The cat sat on the" → "mat" "The cat sat on the mat" → "and" "The cat sat on the mat and" → "purred"
Everything from ChatGPT, Claude, or Gemini is next-token prediction repeated thousands of times. The magic is in the scale.
💡 Analogy

The World's Best Autocomplete

Your phone keyboard predicts the next word. An LLM is the same concept trained on trillions of words — giving it near-human ability to continue any text.

Phone: "How are" → "you"

LLM: "Explain quantum entanglement to a 5-year-old" → coherent multi-paragraph explanation.

How LLMs Work — Under the Hood

⚙ Architecture

LLMs use the decoder-only Transformer from the previous lecture:

// GPT architecture: Input tokens → Embeddings + Position → Masked Self-Attention (causal) → Feed-Forward Network → Add & Norm → Repeat × 96 layers → Linear + Softmax → next token
Decoder-only: No encoder. The model only generates.
Massive scale: GPT-3 has 96 layers, 96 heads, d=12,288.
Autoregressive: One token at a time, feeding each output back as input.
💡 Analogy

Writing a Story One Word at a Time

Imagine writing a novel where you can only see what you've written — never the ending. At each word: "Given everything so far, what's the best next word?"

Because it trained on so much text, its "best guesses" create coherent paragraphs, arguments, and code.

💡 LLMs don't "think." They are sophisticated pattern matchers that absorbed the statistical structure of human language.

Training Pipeline — 3 Stages

⚙ From Raw Internet to ChatGPT
Stage 1: Pre-training

Train on massive text. Learn grammar, facts, reasoning by predicting next token trillions of times.

Data: Books, Wikipedia, web, code
Cost: $2M–$100M+
Output: Foundation model
Stage 2: Fine-Tuning (SFT)

Train on curated (prompt, ideal_response) pairs. Model learns to follow instructions.

Data: ~100K quality examples
Cost: Human labelers
Output: SFT model
Stage 3: RLHF

Humans rank outputs. Reward model learns preferences. LLM optimized for human preference.

Data: Ranked responses
Method: PPO reinforcement learning
Output: Aligned LLM
RLHF separates GPT-3 from ChatGPT. GPT-3 completes text but can be toxic. RLHF teaches it to be conversational, safe, and useful.

What LLMs Can and Cannot Do

✔ What to Expect
Articulation — fluent, well-structured writing in any style
General knowledge — broad understanding across topics
Creativity — brainstorming, diverse perspectives
Translation — between languages, registers, formats
Code generation — writing, debugging, explaining
Summarization — condensing long documents
🚫 What NOT to Expect
Consistent accuracy — may state wrong facts confidently
Domain expertise — no deep niche field knowledge
Mathematics — arithmetic and multi-step reasoning
Real-time info — knowledge has a training cutoff
Guardrails — can be manipulated into harmful output
Source citation — may fabricate references

Hallucinations — When LLMs Lie Confidently

⚙ What Are Hallucinations?

Text that sounds plausible and is well-articulated but is factually incorrect.

❌ Hallucination Example
Prompt: Who first walked on the moon? LLM: Buzz Aldrin is widely recognized as the first person to walk on the moon... // Confidently wrong! It was Neil Armstrong.

LLMs optimize for plausibility, not truth. They pick tokens that sound right — no fact-checking built in.

Mitigation: Verify facts. Use RAG (next week!). Lower temperature. Ask for sources then check them.
💡 Analogy — The Confident Bluffer

A student who hasn't studied but speaks perfectly. Confident tone, perfect grammar — but making it all up. You'd never guess they're wrong from how they sound.

LLMs learned the pattern of correct answers, not the actual facts.

Prompt Engineering — Talking to LLMs

⚙ What is Prompt Engineering?

Prompt: Instructions sent to an LLM to accomplish a task.

Engineering: Iteratively refining that prompt until the output is right.

❌ Vague Prompt
Tell me about dogs.
✔ Engineered Prompt
Write a 200-word blog post about the top 3 health benefits of owning a dog, targeting first-time pet owners, in a warm conversational tone. Include one statistic per benefit.
Same model, same topic — but the engineered prompt controls behavior, constrains output, and improves quality.
💡 Ordering at a Restaurant

Bad: "Give me food." — you'll get something.

Good: "Grilled salmon, medium-rare, steamed veg instead of fries, no sauce, gluten-free." — you get exactly what you want.

The kitchen (LLM) is talented but needs clear, specific instructions.

LLM Parameters — Tuning the Output

⚙ Sampling Parameters — Control HOW tokens are selected
🌡 Temperature
Controls randomness by scaling the logits before softmax. Lower = sharper distribution (more deterministic). Higher = flatter distribution (more creative/random).
0.0 = greedy (always picks #1 token)
0.7 = balanced (default for most APIs)
1.0 = full distribution
1.5+ = very random, may lose coherence
🎯 Top P (Nucleus Sampling)
Only considers the smallest set of tokens whose cumulative probability exceeds P. Cuts off the long tail of unlikely tokens dynamically.
top_p=0.1 → only top ~10% probability mass
top_p=0.9 → considers most tokens (default)
top_p=1.0 → considers all tokens
Use either temp OR top_p, not both
🎲 Top K
Only considers the K most probable next tokens. Hard cutoff — simpler than Top P. Used in Gemini, Claude, and open-source models.
top_k=1 → greedy (same as temp=0)
top_k=10 → choose from top 10 tokens
top_k=40 → moderate diversity (common)
top_k=0 → disabled (consider all)
🎢 Seed
Sets the random number generator seed. Same seed + same prompt = same output. Essential for reproducible experiments and testing.
seed=42 → deterministic output
seed=None → different each time
Not all APIs support this yet
⚙ Output Control Parameters
📏 Max Tokens
Maximum number of tokens the model will generate. Does NOT guarantee that length — it's a ceiling. 1 token ≈ ¾ of a word in English.
50 → short answer
500 → paragraph
4096 → full essay
128,000 → GPT-4o context window
⏹ Stop Sequences
Generation stops immediately when any stop sequence is produced. Prevents the model from rambling or generating past a natural boundary.
stop=["\\n"] → stop at newline
stop=["END"] → stop at custom marker
stop=["\\n\\n", "###"] → multiple triggers
Critical for structured extraction
💬 Echo
Returns the prompt alongside the completion in the API response. Useful for debugging — see exactly what the model received as input.
echo=true → response includes prompt
echo=false → only new tokens (default)
OpenAI completions API only
Chat API uses messages instead
📊 Logprobs
Returns the log probability of each generated token and its top alternatives. Essential for understanding model confidence and debugging.
logprobs=5 → show top 5 alternatives
logprobs=null → disabled (default)
High logprob = model is confident
Low logprob = model is guessing
⚙ Repetition & Penalty Parameters
🔄 Frequency Penalty
Penalizes tokens proportionally to how often they've appeared. Higher = stronger penalty on repeated words. Reduces exact repetition.
0.0 = no penalty (may repeat a lot)
0.5 = light penalty
1.0 = moderate (good default)
2.0 = strong (very diverse vocabulary)
💡 Presence Penalty
Penalizes tokens that have appeared at all, regardless of count. Encourages the model to introduce new topics rather than staying on one.
0.0 = no penalty
0.6 = encourages topic diversity
2.0 = strongly pushes new topics
Frequency = how often. Presence = at all.
📚 System Message
Sets the model's persona, rules, and constraints for the entire conversation. Persists across all turns. The most powerful prompting lever.
role: "system"
content: "You are a helpful medical assistant. Never give diagnoses. Always recommend consulting a doctor."
📋 Response Format
Forces structured output format. GPT-4 supports JSON mode. Guarantees parseable output for programmatic use.
response_format={"type":"json_object"}
Must also instruct "respond in JSON"
in the prompt itself
TaskTemperatureTop P/KOther
Code generation0.0–0.2top_p=0.1stop=["```"]

Creative writing0.8–1.0top_p=0.9presence_penalty=0.6

Golden rule: Don't use Temperature and Top P together — they both control randomness and can interfere. Pick one. For most use cases, Temperature alone is sufficient.

Prompt Engineering Techniques

⚙ Foundational Techniques
1. Zero-Shot
Direct instruction, no examples
Classify sentiment as positive/negative/neutral: "The food was great but service was slow."
2. Few-Shot
Provide examples first
"Amazing product!" → Positive "Terrible." → Negative "It was okay." → Neutral "Great food but slow service." →
3. Chain-of-Thought (CoT)
Step-by-step reasoning
47 apples, sell 23, receive 35. How many? Think step by step: 47 - 23 = 24, then 24 + 35 = 59
4. Role Prompting
Assign a persona
You are a senior data scientist at a bank. Explain why XGBoost beats logistic regression for fraud detection.
5. Constrained Output
Specify exact format
Extract entities. Respond ONLY in JSON: {"persons":[], "locations":[], "orgs":[]} "Satya Nadella said Microsoft will open a new office in Toronto."
💡 When to Use Each
TechniqueBest For
Zero-shotSimple, well-defined tasks
Few-shotTasks needing specific format
Chain-of-thoughtMath, logic, reasoning
Role promptingDomain-specific responses
ConstrainedStructured output (JSON, CSV)
💡 Combine them! "You are a medical coder. Here are 3 examples. Respond only in ICD-10 codes."

Advanced Prompt Techniques

⚙ 6. ReAct (Reasoning + Acting)

The model alternates between thinking (reasoning) and doing (taking actions like searching, computing, or calling APIs). Proposed by Yao et al., 2022.

Thought → Action → Observation loop
Question: What is the population of the capital of Canada? Thought 1: I need to find the capital of Canada first. Action 1: Search("capital of Canada") Observation 1: Ottawa is the capital of Canada. Thought 2: Now I need Ottawa's population. Action 2: Search("population of Ottawa") Observation 2: Ottawa has ~1.07 million people. Thought 3: I have the answer now. Action 3: Finish("~1.07 million")
Why it matters: Pure CoT can hallucinate facts. ReAct grounds reasoning in real observations by calling tools between reasoning steps. It's how AI agents work.
💡 Analogy — The Research Student

Think, then Google, then Think Again

A student solving a homework problem doesn't just think in their head until they get an answer. They:

1. Think: "I need to know Canada's capital"
2. Act: Google it → find "Ottawa"
3. Observe: Read the result
4. Think: "Now I need Ottawa's population"
5. Act: Google that too
6. Answer: with confidence

ReAct teaches LLMs to do exactly this — interleave reasoning with real-world actions instead of guessing everything from memory.

⚙ 7. Self-Consistency

Run the same CoT prompt multiple times with temperature > 0, then take the majority vote of the final answers. Different reasoning paths may reach different conclusions — the most common one is usually correct.

Same prompt, 5 runs, majority wins
// Q: "Is 17 prime?" — run 5 times with temp=0.7 Run 1: 17/2=8.5, 17/3=5.6... no divisors → Yes Run 2: Check 2,3,4...16 → Yes Run 3: sqrt(17)≈4.1, check 2,3 → Yes Run 4: 17=1×17 only → Yes Run 5: Odd, not div by 3... → Yes Majority vote: Yes (5/5) ✔
When to use: Math problems, logic puzzles, code debugging — anywhere a single CoT might take a wrong path. Costs 5× more tokens but dramatically improves accuracy.
💡 Analogy — Ask 5 Friends

If you're unsure about a math answer, ask 5 friends to solve it independently. If 4 out of 5 say "59", and one says "61", you go with 59.

Each friend might take a different approach (algebra, estimation, calculator), but the majority answer is almost always correct. That's self-consistency — wisdom of the crowd applied to reasoning paths.

⚙ 8. Tree of Thought (ToT)

Instead of one linear chain of reasoning, explore multiple reasoning branches in parallel, evaluate each branch, then continue only the most promising ones. Like a chess player thinking several moves ahead.

// Problem: "Arrange 4, 7, 8, 3 to make 24" Branch A: 4 × 7 = 28... 28 - 3 = 25 ❌ (close but wrong) Branch B: 8 × 3 = 24... done! ✔ (8 × 3 = 24) Branch C: 7 + 3 = 10... 10 × 4 = 40 ❌ (too high) // Evaluator picks Branch B → 8 × 3 = 24 ✔ // Prune A and C, expand B
Key difference from CoT: CoT = one path forward. ToT = explore a tree of paths, evaluate, prune bad branches, backtrack if needed. Much more powerful for puzzles and planning.
💡 Analogy — Chess Player

A beginner chess player thinks one move ahead (CoT). A grandmaster thinks 5 moves ahead on 3 different lines, mentally evaluates each position, and picks the most promising path.

Tree of Thought gives LLMs that grandmaster-level planning ability: branch, evaluate, prune, select.

⚙ 9. Reflexion

The model critiques its own previous output, identifies what went wrong, and generates an improved version. A self-improvement loop without any external feedback. Proposed by Shinn et al., 2023.

Generate → Reflect → Improve
Task: Write a SQL query to find customers who spent more than $1000 last month. Attempt 1: SELECT * FROM orders WHERE amount > 1000; Reflection: This query has issues: - No date filter for "last month" - No GROUP BY customer (sums not checked) - Should use SUM not individual amounts Attempt 2 (improved): SELECT customer_id, SUM(amount) as total FROM orders WHERE order_date >= DATE_SUB(NOW(), INTERVAL 1 MONTH) GROUP BY customer_id HAVING total > 1000;
How it works: The model acts as both the student AND the teacher. Generate → critique → regenerate. Each iteration gets closer to correct. Works especially well for code, writing, and logical reasoning.
💡 Analogy — The Self-Grading Student

After finishing an essay, you re-read it yourself with fresh eyes. You notice a weak paragraph, a missing argument, and a typo. You rewrite those parts.

Reflexion is this self-editing loop: the LLM generates, then asks itself "what's wrong with this?", then fixes the issues. No external reviewer needed — the model becomes its own critic.

⚙ 10. Prompt Chaining

Break a complex task into sequential sub-tasks, where each step's output feeds into the next step's input. Each link in the chain is a focused, simpler prompt.

// Complex task: "Summarize this article and translate to French" Chain Step 1: "Extract the 5 key points from this article:" → Output: [5 bullet points] Chain Step 2: "Write a 100-word summary from these points:" → Output: [English summary] Chain Step 3: "Translate this summary to French:" → Output: [French summary] // Each step is simple. The chain handles complexity.
Why chain? A single prompt asking "summarize and translate this 5000-word article to French" often fails. Breaking it into 3 focused steps gives dramatically better results — each step does one thing well.
💡 Analogy — Assembly Line

A car isn't built by one person doing everything. It moves through stations: frame → engine → wiring → paint → interior. Each station specializes in one task and passes the result forward.

Prompt chaining is the same: each prompt is a station. Complex tasks that overwhelm a single prompt become manageable when decomposed into focused steps.

LLM Ethics

⚙ Key Concerns
⚠ Bias Amplification

LLMs learn from internet data containing societal biases. They can amplify stereotypes.

⚠ Misinformation at Scale

Can generate convincing fake news and fabricated citations at unprecedented speed.

⚠ Privacy & Data Leakage

May memorize and reproduce personal information from training data.

💡 The Responsible Approach
1. Audit outputs — test for bias across demographics
2. Human-in-the-loop — don't fully automate critical decisions
3. Transparency — disclose AI-generated content
4. Guardrails — content filters and safety layers
5. Diverse training — inclusive, representative datasets
The question isn't "can we build this?" but "should we, and how do we do it responsibly?"
