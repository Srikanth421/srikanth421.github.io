Why Do We Need RAG?

⚙ The Problem RAG Solves

LLMs have two fundamental limitations:

❌ Problem 1 — Knowledge Cutoff
User: How many CUDA cores does the NVIDIA RTX 5070 Ti have? LLM: I don't have information on that, as my training data goes up until 2023.
❌ Problem 2 — No Private Data
User: What is our company's refund policy? LLM: I don't have access to your company's specific policies.
✔ RAG Solution
User: How many CUDA cores does the RTX 5070 Ti have? RAG retrieves: [NVIDIA spec sheet chunk about RTX 5070 Ti] LLM + Context: The RTX 5070 Ti has 8960 CUDA cores with a boost clock of 2300 MHz.
💡 Analogy — Open-Book Exam

From Closed-Book to Open-Book

A plain LLM is a closed-book exam — answers only from memory. RAG turns it into an open-book exam — the student can find the relevant page and base their answer on it.

RAG = Retrieval + LLM. The LLM's ability stays the same. We just give it the right information at the right time.
📚 Professor Notes — Why Not Just Fine-Tune?
Fine-TuningRAG
Changes model weightsLeaves model unchanged
Expensive retraining ($$$)Cheap indexing
Static knowledgeDynamic — re-index anytime
Hard to updateUpdate = re-index one PDF
Risk of catastrophic forgettingNo forgetting

Example: Bank updates mortgage policy today.

Fine-tuning: Retrain model → redeploy → hours/days.

RAG: Re-index the PDF → done → minutes.

Hallucination reduction formula: Without RAG, the model computes P(answer | question). With RAG, it computes P(answer | question, retrieved_documents). The model now has evidence to work with.

RAG Architecture — The Full Pipeline

⚙ The Pipeline (Two Phases)

Indexing (prepare docs, done once) and Querying (answer questions, per query).

// ━━━ INDEXING PHASE (done once) ━━━━━━━━━━━━━━━━━━ ① LOAD PDF, TXT, HTML, CSV → raw text ↓ ② CHUNK Split into 200-500 token pieces (with overlap) ↓ ③ EMBED Each chunk → vector [0.12, 0.85, 0.33, ...] ↓ ④ STORE Vectors → Vector Database // ━━━ QUERY PHASE (per user question) ━━━━━━━━━━━ User asks: "What is the RTX 5070 Ti boost clock?" ↓ EMBED query → vector RETRIEVE top-K closest chunks from Vector DB AUGMENT stuff chunks into the LLM prompt GENERATE LLM answers using context → "2300 MHz"
Correct pipeline sequence: Data Chunk → Embedding → Vector DB → Retriever → LLM
📚 Professor Notes — Computational Complexity
// Why two phases? 10,000 documents × 500 chunks each = 5,000,000 chunks Brute-force search: O(N) per query → Too slow for production! ANN (Approximate Nearest Neighbor): O(log N) per query ✔ Expensive work (indexing) done once. Cheap work (querying) done per question.

This is why we split into two phases:

Indexing cost: O(N) — done once, amortized.

Query cost: O(log N) — fast, per question.

This also motivates vector databases and ANN algorithms (HNSW, IVF, Product Quantization) — they trade a tiny accuracy loss for massive speed gains.

Building Blocks of RAG

⚙ The 5 Components
📄 Document Loader

Ingests PDFs, websites, databases, APIs. Libraries: LangChain, LlamaIndex, Unstructured.io.

✂ Text Splitter / Chunker

Breaks docs into smaller pieces. Controls chunk size and overlap.

🔬 Embedding Model

Converts chunks into vectors. OpenAI, Cohere, MiniLM, BGE.

🗃 Vector Database

Stores vectors for similarity search. Pinecone, ChromaDB, Weaviate, FAISS, pgvector.

🤖 LLM (Generator)

Takes query + retrieved context, generates final answer. GPT-4, Claude, Gemini, Llama.

💡 The Research Assistant

Loader = collecting textbooks
Chunker = cutting into index cards
Embedder = organizing cards by meaning
Vector DB = the filing cabinet
LLM = the writer who reads cards and writes answers

No single component makes RAG work. It's the pipeline that matters.
📚 Professor Notes — Metadata Matters

Every chunk should carry metadata. Vector DBs store embeddings + text + metadata, not just vectors:

// Chunk metadata example: { "source": "HR_policy.pdf", "page": 12, "section": "Vacation Policy", "last_updated": "2025-01-15" } // Enables filtered search: db.query(q, filter={"source": "HR_policy.pdf"})

Retriever vs Generator — two different problems:

🔎 Retriever answers: "Where should I look?"

📝 Generator answers: "How do I explain it?"

Critical insight: Most RAG failures are NOT LLM failures. They are retrieval failures. Wrong chunk in → wrong answer out. The LLM is innocent. Always debug retrieval first.

Chunking — Breaking Documents Into Pieces

⚙ Chunk Size and Overlap
// chunk_size=512, chunk_overlap=16 Chunk 1: tokens 1–512 Chunk 2: tokens 497–1008 ← 16 tokens overlap Chunk 3: tokens 993–1504 ← 16 tokens overlap // Why overlap? Information might span a boundary. // "The patient was diagnosed with | type 2 diabetes" // Without overlap, Chunk 1 is incomplete.
StrategyHowBest For
Fixed-sizeEvery N tokens + M overlapSimple, general
SentenceSplit on sentence boundariesPreserves thoughts
RecursiveParagraphs → sentences → charsMost popular
SemanticSplit when embedding similarity dropsTopic-based
DocumentBy headers/sections/pagesManuals, legal
Sweet spot: 256–512 tokens, 10–20% overlap. Test for your data.
💡 Cutting a Novel Into Index Cards

Too small (50 tokens) = fragmented, missing context.
Too large (2000 tokens) = too much noise, dilutes signal.

Overlap is like copying the last sentence of each page onto the next card — continuity preserved.

📚 Professor Notes — Chunking Deep Dive

Precision vs Recall Tradeoff

Small chunks → High precision, low recall (very focused but may miss context).
Large chunks → High recall, low precision (captures more but adds noise).

⚠ Lost in the Middle Problem

LLMs struggle with info buried deep in long contexts. If you retrieve 20 chunks and the relevant one is ranked #12, the model may ignore it. Recent research (Liu et al., 2023) confirmed this — models attend most to the beginning and end of context.

Embeddings — Text to Vectors

⚙ Why Embeddings? (Callback to Week 3)

We embed entire chunks into vectors to search by meaning, not keywords.

// Why not keyword search? Query: "heart attack symptoms" Doc: "myocardial infarction clinical presentation" Keyword match: 0 words overlap ❌ Semantic match: cosine sim = 0.89 ✔ // Embeddings capture MEANING embed("heart attack symptoms") = [0.82, 0.15, 0.71, 0.33] embed("myocardial infarction signs") = [0.80, 0.18, 0.69, 0.35] cosine_similarity = 0.89 embed("stock market crash") = [0.11, 0.72, 0.05, 0.88] cosine_similarity = 0.12
ModelProviderDimsNotes
text-embedding-3-smallOpenAI1536Good default
all-MiniLM-L6-v2HuggingFace384Free, local
BGE-large-enBAAI1024Open-source SOTA
Vertex AI embeddingsGoogle768GCP native
Titan EmbeddingsAWS1536Bedrock native
💡 Same Concept, New Scale

Week 3: Word2Vec embeds words. RAG embeds paragraphs. Same principle: similar meanings → nearby vectors.

The embedding model is the most important RAG choice. Bad embeddings = bad retrieval = bad answers.

📚 Professor Notes — Cosine Similarity Math

Students should know this formula:

Cosine(A, B) = (A · B) / (|A| × |B|) Range: [-1, 1] For text embeddings: usually [0, 1] 1.0 = identical meaning 0.0 = unrelated -1.0 = opposite meaning (rare in practice)

Why Cosine, Not Euclidean Distance?

Cosine ignores vector magnitude, only measures direction.

[1, 2, 3] and [10, 20, 30] Euclidean distance: 37.4 (far apart!) Cosine similarity: 1.0 (same direction!) // Same meaning, different scale. // Cosine captures semantic direction. // Euclidean is misled by magnitude.

💡 The Smart Library

Regular DB = sorted alphabetically. Vector DB = sorted by meaning. Ask for "a language for data science" and it finds Python, R, Julia — no keyword match needed.

📚 Professor Notes — ANN Search Algorithms

Exact nearest neighbor: O(N) — check every vector. ANN trades tiny accuracy loss for massive speed:

AlgorithmUsed ByHow It Works
HNSWPinecone, Qdrant, WeaviateMulti-level graph. Search starts high, zooms down.
IVFFAISSCluster vectors, only search nearby clusters.
PQFAISSCompress vectors. Less memory, fast comparison.

HNSW Intuition

Imagine finding a restaurant in Toronto:

Level 3: Canada → Level 2: Ontario → Level 1: Toronto → Level 0: exact neighborhood.

Start at the top, zoom down. Skip millions of irrelevant vectors. This is how HNSW achieves O(log N) search.

Semantic Search vs Keyword Search

⚙ The Core Innovation
❌ Keyword Search
Query: "How to fix a broken heart" 1. "Heart repair surgery" ❌ 2. "Broken glass cleanup" ❌ // Matches WORDS, not meaning
✔ Semantic Search
Query: "How to fix a broken heart" 1. "Coping with emotional pain" 0.91 2. "Moving on after breakup" 0.87 // Understands INTENT
Best practice: Hybrid search — combine semantic (vector) with keyword (BM25). Weaviate and Pinecone support this natively.
📚 Professor Notes — Hybrid Search Formula
// Production hybrid search formula: Score = α × BM25(query, doc) + (1-α) × Cosine(q_vec, doc_vec) where α = 0.3 to 0.5 typically BM25 = keyword relevance (exact matches) Cosine = semantic relevance (meaning)

⚠ When Semantic Search Fails

Query: "Apple" Could mean: • The fruit 🍎 • The company 🍏 Embeddings may retrieve wrong context. Keyword filters help disambiguate.

This is why pure semantic search is rarely used alone in production. Hybrid search catches what embeddings miss.

Worked Example — HR Policy Bot

⚙ End-to-End Pipeline
// ━━━ INDEXING ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ Load: docs = load_pdf("company_handbook.pdf") # 200 pages Chunk: chunks = split(docs, size=512, overlap=50) # 847 chunks Embed: vectors = model.encode(chunks) # 847 x 1536 Store: db.upsert(vectors, chunks, metadata) # indexed // ━━━ QUERYING ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ User: "How many sick days do I get per year?" Embed: q_vec = model.encode(query) Retrieve: top_3 = db.query(q_vec, top_k=3) # sim: 0.92, 0.88, 0.81 Augment: prompt = "Answer based ONLY on context below. Context: [retrieved chunks] Question: How many sick days?" Generate: "Full-time employees receive 12 paid sick days per year." // Grounded in actual company policy ✔
The LLM never saw the handbook during training. RAG injected relevant chunks at query time. No hallucination possible when the answer comes from your own documents.

LangChain Code Pattern — What You'll Use in the Case Study

⚙ Complete LangChain RAG Pipeline (Python)
# ━━━ STEP 1: LOAD PDF ━━━━━━━━━━━━━━━━━━━━━━━━━━ from langchain_community.document_loaders import PyMuPDFLoader pdf_loader = PyMuPDFLoader("apple_hbr.pdf") apple = pdf_loader.load() # returns list of Document objects len(apple) # → 11 pages # ━━━ STEP 2: CHUNK ━━━━━━━━━━━━━━━━━━━━━━━━━━━━ from langchain.text_splitter import RecursiveCharacterTextSplitter text_splitter = RecursiveCharacterTextSplitter.from_tiktoken_encoder( encoding_name='cl100k_base', # OpenAI's tokenizer chunk_size=512, # tokens per chunk chunk_overlap=20 # overlap between chunks ) chunks = pdf_loader.load_and_split(text_splitter) len(chunks) # → ~25-30 chunks from 11 pages # ━━━ STEP 3: EMBED ━━━━━━━━━━━━━━━━━━━━━━━━━━━━ from langchain_community.embeddings import SentenceTransformerEmbeddings embedding_model = SentenceTransformerEmbeddings( model_name='thenlper/gte-large' # 1024-dim vectors ) # ━━━ STEP 4: STORE IN CHROMADB ━━━━━━━━━━━━━━━━━ from langchain_community.vectorstores import Chroma vectorstore = Chroma.from_documents( chunks, embedding_model, persist_directory="apple_db" # saves to disk ) # ━━━ STEP 5: CREATE RETRIEVER ━━━━━━━━━━━━━━━━━━ retriever = vectorstore.as_retriever( search_type='similarity', search_kwargs={'k': 3} # return top 3 chunks )
cl100k_base is OpenAI's tokenizer (used by GPT-4). Using it for chunking ensures token counts match what the LLM will see. gte-large is a high-quality open-source embedding model (1024 dims).

Running LLMs Locally — No API Needed

⚙ GGUF Models + llama-cpp-python

You don't need OpenAI or cloud APIs. Open-source models like Mistral 7B can run on your laptop using quantized GGUF format:

# Download quantized Mistral 7B from HuggingFace from huggingface_hub import hf_hub_download from llama_cpp import Llama model_path = hf_hub_download( repo_id="TheBloke/Mistral-7B-Instruct-v0.2-GGUF", filename="mistral-7b-instruct-v0.2.Q6_K.gguf" ) # Load model (GPU) llm = Llama( model_path=model_path, n_ctx=5000, # context window (tokens) n_gpu_layers=38, # layers offloaded to GPU n_batch=512 ) # For CPU only: # llm = Llama(model_path=model_path, n_ctx=1024, n_cores=-2)
💡 What is GGUF?

GGUF (GPT-Generated Unified Format) is a file format for storing quantized LLMs. Quantization shrinks a model from 14GB (full precision) to ~5GB (Q6_K) by using fewer bits per weight.

Q6_K means 6-bit quantization — nearly full quality at 40% the size. Q4_K_M is even smaller but slightly less accurate.

This lets you run a 7-billion parameter model on a laptop with 8GB RAM. No cloud API, no costs, fully private.

Prompt Template Design for RAG

⚙ System + User Prompt Structure

The prompt template is how you inject retrieved context into the LLM. Two parts: a system message (persona + rules) and a user message (context + question).

System Prompt — Set the Rules
system_message = """ You are an assistant whose work is to review the report and provide appropriate answers from the context. User input will have context beginning with: ###Context The context contains references to specific portions of a document relevant to the user's question. User input will also have a question beginning with: ###Question Please answer ONLY using the context provided. Do not use any prior knowledge or information. If the answer is not found in context, respond: "I could not find the answer in the provided context." """
User Prompt — Context + Question
user_message = """ ###Context Here are some documents relevant to the question: {context} ← retrieved chunks injected here ###Question {question} ← user's actual question """
The "answer ONLY from context" instruction is critical. Without it, the LLM will mix retrieved facts with its own training knowledge — defeating the purpose of RAG. This is what prevents hallucinations.

RAG Prompt Design — The Template Pattern

⚙ System + User Prompt Template

A RAG prompt has two parts: a system message defining the assistant's behavior and a user message injecting the retrieved context.

System Prompt (set once)
You are an assistant whose work is to review the report and provide appropriate answers from the context. User input will have context beginning with the token: ###Context. The context contains references to specific portions of a document. If the answer to the question cannot be derived from the context, say you don't have enough information.
User Prompt Template (per query)
###Context Here are documents relevant to the question: {retrieved_chunks} ###Question {user_question}
Key design choice: The system prompt explicitly tells the model NOT to answer if the context doesn't contain the information. This is how RAG eliminates hallucinations — the model is constrained to the retrieved evidence.
💡 Why ###Context and ###Question Tokens?

These delimiter tokens (###Context, ###Question) act as clear section markers that help the LLM distinguish between the retrieved evidence and the actual question.

Without delimiters, the model might confuse context text with instructions. With them, it knows exactly: "everything between ###Context and ###Question is my evidence; everything after ###Question is what I need to answer."

Common delimiter patterns:
###Context ... ###Question (case study pattern) <context> ... </context> (XML-style) --- BEGIN CONTEXT --- (separator-style) [INST] ... [/INST] (Mistral/Llama format)
📚 Professor Notes — Context Window Budget
// Model limit: 32,000 tokens // You must fit EVERYTHING in this budget: System prompt: ~200 tokens Retrieved chunks: k × 512 = ~1,536 tokens (k=3) User question: ~50 tokens Answer space: ~500 tokens ———————————————— Total: ~2,286 tokens (fits!) But if k=20: 20 × 512 = 10,240 tokens → Less room for the answer!

Citation Prompting (Production RAG)

In banking, healthcare, and legal, RAG must cite sources:

// Add to system prompt: "Always cite the source document and page number for each claim in your answer." // LLM output: "Full-time employees receive 12 sick days per year (HR_Policy.pdf, page 12)."

This is why metadata matters — the LLM can only cite what the retriever passes along.

RAG Parameter Tuning

⚙ The Parameters That Matter Most

RAG quality depends on tuning both retrieval and generation parameters:

ParameterEffectToo LowToo High
k (retrieval)Number of chunks retrievedMay miss the answer entirelyAdds noise, dilutes context
chunk_sizeTokens per chunkFragmented, missing contextToo broad, irrelevant text
chunk_overlapShared tokens between chunksMay split key sentencesRedundancy, storage cost
max_tokensMax response lengthAnswer gets cut off mid-sentenceModel may pad with noise
temperatureOutput randomnessRigid, may refuse to elaborateMay hallucinate despite context
top_p / top_kToken sampling diversityRepetitive responsesIncoherent responses
// Typical RAG parameter settings: Retrieval: k=3, chunk_size=512, chunk_overlap=20 Generation: max_tokens=256, temperature=0, top_p=0.95, top_k=50 // From the case study — effect of tuning: // Query: "List 3 leadership characteristics" max_tokens=128 → Only 2 characteristics (cut off) max_tokens=350 → All 3 characteristics returned ✔
💡 Tokenization — How Chunk Size Is Measured

Chunk size is measured in tokens, not characters. Different tokenizers produce different token counts for the same text.

// Same sentence, different tokenizers: "The cat sat on the mat" cl100k_base (GPT-4/OpenAI): 6 tokens LLaMA tokenizer: 7 tokens Word-level: 6 tokens // The case study uses tiktoken cl100k_base from tiktoken import encoding_for_model enc = tiktoken.get_encoding("cl100k_base") tokens = enc.encode("The cat sat on the mat") len(tokens) # 6

When you set chunk_size=512, the splitter counts 512 tokens in that specific tokenizer, not 512 characters or 512 words. Always match the tokenizer to your embedding/LLM model.

// THE CASE STUDY STACK: PDF Loader: PyMuPDFLoader (langchain + pymupdf) Chunker: RecursiveCharacterTextSplitter .from_tiktoken_encoder( encoding="cl100k_base", chunk_size=512, chunk_overlap=20) Embeddings: SentenceTransformer("thenlper/gte-large") 1024 dimensions, runs locally Vector DB: ChromaDB (local, persisted to disk) LLM: Mistral-7B-Instruct-v0.2 (GGUF) via llama-cpp-python GPU: n_gpu_layers=38 CPU: n_cores=-2
GGUF format: A quantized model format that lets you run 7B-parameter models on a laptop. The Q6_K variant balances quality and speed. Smaller quantizations (Q4) are faster but less accurate.
💡 PDF Extraction Gotcha

Not All Text Extracts Cleanly

When a PDF page has complex layouts (tables, multi-column text, diagrams with labels), the text extractor may not read content in the correct order.

// Page 6 of the Apple HBR report: // Contains shapes, arrows, and text boxes Extracted text is NOT sequential ❌ // Labels from different boxes get mixed together // Solutions: 1. Use layout-aware extractors (Unstructured.io) 2. Use OCR + vision models (GPT-4V on page images) 3. Pre-process PDFs to clean up extraction

This is a real-world limitation. Always inspect your extracted text before chunking — garbage in, garbage out.

RAG Evaluation — LLM-as-a-Judge

⚙ The RAG Triad + LLM-as-a-Judge
🔎
Clarity

Is the prompt clear enough for the LLM?

🙌
Groundedness

Is every claim in the response derived from the context?

🎯
Relevance

Does the response answer what was asked?

LLM-as-a-Judge uses the same LLM to evaluate its own RAG outputs. Give it the question, context, and answer — ask it to rate on a 1–5 scale:

Groundedness Evaluation Prompt
###Question {question} ###Context {retrieved_context} ###Answer {generated_answer} Rate the groundedness of the answer on a scale of 1-5: 1 = answer is completely fabricated 5 = every claim is directly from context Provide your rating and justification.
From the case study: Query 1 ("Who are the authors?") scored 5/5 on both groundedness and relevance — a perfect RAG response. Query 3 about innovation examples correctly refused to answer when context didn't contain the information.
💡 Evaluation Tools
ToolApproach
LLM-as-a-JudgeUse the same or different LLM to rate outputs (case study approach)
RAGASOpen-source framework: faithfulness, relevance, context precision
TruLensTracks groundedness, relevance, and answer quality over time
LangSmithLangChain's evaluation + tracing platform
Human evalManual review — gold standard but doesn't scale

The LLM-as-a-judge approach is fast and cheap but has a flaw: the model might be biased toward rating its own outputs favorably. Using a different, stronger model as judge (e.g., GPT-4 judging Mistral outputs) is more reliable.

📚 Professor Notes — Evaluate Each Stage Separately

RAG Evaluation Pyramid

// Evaluate each layer independently: Layer 1: Retrieval Context Precision — are chunks relevant? Context Recall — do chunks contain answer? Recall@K = relevant_retrieved / relevant_total Layer 2: Generation Faithfulness — answer matches context? Relevance — answer matches question? // Debug bottom-up. Fix retrieval first.

⚠ Example Failure Analysis

Question: "What is mortgage prepayment penalty?" Retrieved chunk: "HELOC rates were adjusted..." → Retriever failed. Wrong chunk. → Answer is wrong. → But the LLM is innocent! // Fix: better embeddings, adjust chunk size, // add keyword filter, increase k
In production, 80% of RAG errors are retrieval errors, not generation errors. Always debug retrieval first.

Advanced RAG Techniques — Where the Industry Is Heading

🚀 Beyond Basic RAG
TechniqueWhat It DoesWhen to Use
Hybrid SearchCombine BM25 keyword search with vector semantic search (α × BM25 + (1-α) × Cosine)Production systems, domain-specific terms
Re-rankingCross-encoder model re-scores top-K retrieved chunks for finer relevanceWhen initial retrieval has high recall but low precision
Multi-Query RAGLLM generates multiple query variations, retrieves for each, merges resultsAmbiguous or complex questions
Parent-Child RetrievalRetrieve small child chunks for precision, return parent chunk for contextWhen small chunks match but lack surrounding context
Graph RAGBuild a knowledge graph from documents, traverse relationships during retrievalMulti-hop reasoning ("Who reports to the CEO's direct reports?")
Agentic RAGLLM decides when/what to retrieve using ReAct-style reasoning loopsComplex tasks requiring multiple retrieval steps
Self-RAGModel decides whether to retrieve, then self-evaluates its own responseAdaptive retrieval — skip retrieval for simple questions
Corrective RAG (CRAG)If retrieved docs are irrelevant, fall back to web search or other sourcesWhen your knowledge base may not have the answer
Knowledge Graph RAGCombine structured knowledge graphs with unstructured vector searchEnterprise data with complex entity relationships
The trend: Basic RAG (retrieve → generate) is becoming the baseline. Production systems layer these techniques on top. Start simple, add complexity only when evaluation metrics demand it.
